# Memory Layer — storage→retrieval upgrade (topic-keyed retrieval)

**Status:** ⬜ Proposed (draft — awaiting `/g-roadmap` slotting; no milestone number assigned yet)
**Version:** ships its own minor when built
**Depends on:** nothing structural — sharpens the existing `/g-resume` retriever and the record-writing skills (`/g-retro`, `/g-adr`). The `## Active Session` git handoff and key+grep retrieval remain the fallback.
**Design note:** the storage/retrieval framing and the RAG mapping live in `g-docs/memory-taxonomy.md`; the "no vector store" trade-off is recorded in ADR-001. This file is the buildable scope.

## Goal
Close the one weak seam in G-Forge's memory layer — **topic-overlap retrieval** — without adding a runtime or a vector store. Today `/g-resume` retrieves Architectural-tier records (ADRs, retros) by *key*: branch slug + recently-touched filenames. A record relevant by *subject* but sharing no slug or filename is missed (e.g. a caching ADR that never named the file you are now editing). The fix is **sharper keys** — topic tags on records plus a light, committed index — so retrieval matches by subject, not just by filename coincidence. This is the in-scope analogue of "semantic retrieval," done with deterministic, auditable keys.

## Why not embeddings
Settled in **ADR-001** (`g-docs/decisions/001-no-vector-store-for-memory-retrieval.md`): no runtime to host a model/index, must travel to web/mobile/Slack/Actions surfaces, small high-signal corpus. The whole point of this milestone is to get most of the topic-relevance gain with zero new runtime and zero external dependency.

## Non-goals
- **No vector store, no embedding model, no external index** — ADR-001 stands; this is keys, not similarity search.
- **No new runtime** — tags are frontmatter; the index is a committed file written by existing skills, not a process.
- **No retroactive backfill required** — pre-existing retros/ADRs without tags still retrieve by today's key+grep path (graceful degradation). Backfill is optional and offered, never forced.

## Scope / tasks

### Phase A — Tag vocabulary + record-writing
- [ ] **A1 — Define a small, controlled topic-tag vocabulary.** Not free-text (free tags drift and never match). A curated set keyed to the project's architecture surface — seeded from the layer map / stack profile — plus a handful of cross-cutting tags (e.g. `auth`, `caching`, `data-flow`, `build`, `testing`, `release`). Document it in `g-docs/memory-taxonomy.md`.
- [ ] **A2 — Add `topics:` frontmatter to records.** `/g-adr` and `/g-retro` prompt for / infer 1–4 topic tags from the controlled vocabulary when writing an ADR or retro, and write them into the file's frontmatter. Closed-vocabulary validation; unknown tag → ask or drop.
- [ ] **A3 — Light committed index.** A single `g-docs/decisions/index.md` (or `g-docs/.memory-index.md`) mapping `topic → [record paths]`, regenerated by the record-writers on each write. Deterministic, diff-able, human-readable — no process owns it beyond the skills that already write the records.

### Phase B — Retrieval
- [ ] **B1 — Teach `/g-resume` topic retrieval.** Step 2 gains a topic-overlap pass: derive the current task's topics (from the first task + touched files' area), look them up in the index, and fold the matches into the candidate set *before* relevance judgment. Key+grep stays as the floor; topic lookup is additive.
- [ ] **B2 — Capped + pointer-rich, unchanged discipline.** Topic matches obey the same cap-and-pointer rule (load distilled sections, leave the rest as `(N more)` pointers). A clean window stays the goal — more *relevant* candidates, not more *loaded* context.

### Phase C — Hygiene + docs
- [ ] **C1 — `/g-doctor` check.** Advisory: records missing `topics:`, tags outside the vocabulary, index drift (index references a moved/deleted record). Offer the fix.
- [ ] **C2 — Optional backfill offer.** A one-time pass that proposes `topics:` for existing untagged records (developer approves each). Never automatic.
- [ ] **C3 — Sync `g-docs/memory-taxonomy.md`, `rules/g-rules/J-memory.md`, README, CHANGELOG.**

## Done condition (DoD)
A retro tagged `topics: [caching]` is retrieved by `/g-resume` when the current task touches caching, **even on a branch whose slug and touched filenames never mention caching** — the case today's key+grep path misses. Untagged records still retrieve by the old path. `/g-doctor` flags an untagged new ADR. No new runtime or external dependency was added (ADR-001 holds).

## Premortem (sketch — firm up at `/g-plan`)
- *Tag vocabulary drift / bikeshedding* (med) — an open vocabulary becomes noise; a rigid one fits no project. → Seed from the project's own layer map + a tiny cross-cutting set; treat it as living, not exhaustive.
- *Index drift* (med) — the index and the records disagree after a manual move. → Regenerate on every record write; `/g-doctor` C1 catches drift; the index is advisory (key+grep is the floor, so drift degrades, never breaks).
- *Tagging burden* (low–med) — prompting for tags on every ADR/retro adds friction. → 1–4 tags, inferred-and-confirmed (not free-typed), from a closed list; skippable.
- *Marginal gain* (low) — if most records already share slugs/filenames with their tasks, topic retrieval adds little. → Validate on the existing corpus during `/g-plan` before committing the full build; if the blind spot is rare, ship only A1–A2 (tags) and defer B.
