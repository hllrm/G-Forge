# M41 — Release Machinery + README Currency

**Status:** ⬜ Not started (approved scope — ready for `/g-plan`)
**Version:** v2.4.0 (minor — new release commands + skills + a `/g-doctor` version-consistency check)
**Goal:** Make cutting a release a single gated step (`/g-changelog` + `/g-release`), and make README/CHANGELOG currency a structural property of every release rather than a memory-dependent chore.

> **RESTRUCTURED 2026-07-18 (developer decision):** the G-Proof rebrand, the full persuasion-ordered README restyle, and field communication were split OUT of this milestone into **`g-docs/milestones/M44.md` — the G-Proof 1.0 capstone, sequenced dead last**. The README/CHANGELOG *currency convention* starts even before this milestone — at the v2.3.0 release (M-audit close ships the first status strip + CHANGELOG cut).

## Context

Six releases in ~2 weeks (2.0.0→2.2.1), each hand-editing the version in three places (`plugin.json`, `marketplace.json`, README counts) + cutting CHANGELOG `[Unreleased]`→dated + (always skipped) tagging. On 2026-07-12 a v2.2.2 bump was made mid-milestone and had to be reverted because nothing gated "is this a coherent, complete release?". On 2026-07-18 the developer flagged the GitHub README as visibly stale. Distribution is straight off `main` — the `plugin.json` version field IS the "latest available" signal every installed project sees, so a wrong bump ships immediately.

## Scope

### Wave 1 — Release Tooling (`/g-changelog`, `/g-release`)

**Tasks:**
1. Design `/g-changelog` skill:
   - Input: the **curated durable record** (milestone-ledger rows, review verdicts, plan done-conditions) as the primary source — **not raw `git log`** (ledger rows are already human-curated signal; commit messages are not). Git history is a cross-reference for confirmation only.
   - Output: draft `## [x.y.z] — date` block with categorized entries (Breaking Changes, Features, Fixes, Docs, Internal)
   - Option to prepend to CHANGELOG.md on approval
   - Gating: human reads + nods before any write
2. Implement `/g-changelog`:
   - Dispatch a `changelog-writer` agent (Haiku-tier) — ledger/review record as source, ADRs + git history for confirmation only, Keep-a-Changelog buckets inferred from row type
   - Return structured block for human review; write only on approval
3. Design `/g-release` skill:
   - Preconditions (hard gates, refuse — don't warn): active milestone ✅ closed; full suite green on a **real run** with pasted evidence (finding #20); gate self-hosted clean; no orphaned `[Unreleased]`; README status strip current (Wave 2)
   - Orchestration: confirm version (developer states major/minor/patch — the skill enforces consistency, never decides the bump) → run `/g-changelog` → one-shot version bump across **every** manifest → `[Unreleased]` → dated `## [x.y.z]` → annotated `v{x.y.z}` tag (closes the tagging gap the alveria adopter works around with pinned SHAs) → final `/g-review` on the release commit → push
   - Output: tag sha + release notes template
4. Implement `/g-release`.
5. Add the `/g-doctor` **version-consistency check**: manifests agree with each other; README counts match the `agents/` + `skills/` + `profiles/` inventory; README status-strip version matches `plugin.json`. Standing backstop against any hand bump.
6. Register both skills in `commands/g-forge.md`; add to README skills table.

**Depends on:** —

**Done condition:** `/g-changelog` drafts an entry from the ledger, human approves, it lands; `/g-release` refuses on a violated precondition (proven with a fixture or live partial-milestone attempt) and completes a full gated cut when preconditions hold; `/g-doctor` check flags a deliberately desynced manifest.

### Wave 2 — README Currency Machinery

**Tasks:**
7. `/g-release` precondition: the README **status strip** (version badge · "What's new" → CHANGELOG.md · "Where this is going" → g-docs/ROADMAP.md — first shipped at v2.3.0 by M-audit's release pass) is present and matches the release version being cut.
8. `/g-review` Step-6 close-out README-currency pass (optional, behavior-change-gated — the reusable per-milestone mechanism; drafted + nodded, never auto-applied).

**Depends on:** Wave 1 (the consistency check it extends)

**Done condition:** cutting a release with a stale strip is impossible via `/g-release`; the close-out pass exists and is documented in the skill.

## Premortem

| Risk | Scenario | Mitigation |
|------|----------|-----------|
| **Rubber stamp** (med) | Preconditions advisory → it just automates a bad bump | Milestone-closed + green-run + strip-current are hard gates; refuse, don't warn |
| **Version drift reappears** (low) | Someone hand-bumps one manifest | `/g-doctor` consistency check is the standing backstop |
| **Tag/manifest divergence** (low) | Tag cut separately from manifest write | Same run writes both; never a separate step |
| **Currency convention decays** (med) | Strip goes stale between releases | Strip check is a `/g-release` precondition, not a habit; doc gate covers CHANGELOG on every mixed/doc commit |
