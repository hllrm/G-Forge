# M41 — Release & Rebranding

**Status:** ⬜ Not started (approved scope — ready for `/g-plan`)
**Version:** v3.0.0 (semver major — product rename + version strategy)
**Goal:** Rebrand G-Forge → G-Proof, ship release machinery (`/g-changelog`, `/g-release`), and rewrite README with persuasion-ordered structure, gate GIF, and positioning narrative.

---

## Context

**Decision:** rebrand to **G-Proof** (Gianmarco's shipping standard for Claude Code). The G-Factor is the human in the loop; the gate enforces discipline before merge. Tagline: **"Claude Code is powerful. It's also optimized for velocity, not reliability. But you can G-Proof it."** — positioning around governance layer (educated, enforced project management), not another agent orchestrator.

**Scope:** rebrand + version bump + release tooling + README rewrite + field communication. All folded into one milestone to avoid re-announcing.

### Wave 2 — Release Tooling (`/g-changelog`, `/g-release`)

**Tasks:**
7. Design `/g-changelog` skill:
   - Input: the **curated durable record** (milestone-ledger rows, review verdicts, plan done-conditions) as the primary source — **not raw `git log`** (ledger rows are already human-curated signal; commit messages are not). Git history is a cross-reference for confirmation only.
   - Output: draft `## [3.0.0] — 2026-MM-DD` block with categorized entries (Breaking Changes, Features, Fixes, Docs, Internal)
   - Option to prepend to CHANGELOG.md on approval
   - Gating: human reads + nods before any write
8. Implement `/g-changelog` skill:
   - Dispatch to a `changelog-writer` agent (Haiku-tier) — read the ledger/review record as the source, cross-ref ADRs + git history for confirmation only, infer Keep-a-Changelog buckets from row type, propose entries
   - Return structured block for human review
   - Write only on approval
9. Design `/g-release` skill:
   - Input: approved plan (exists — M41 itself)
   - Orchestration: confirm version + tag + branch; run `/g-changelog`; run final `/g-review` on the release commit; push tag + branch; trigger marketplace sync (if available)
   - Output: release notes + confirm URL
10. Implement `/g-release` skill:
    - Validation gates (version matches package manifests, CHANGELOG entry exists)
    - Tag creation: `git tag -a v3.0.0 -m "G-Proof v3.0.0: Rebranding + release machinery"`
    - Optional: draft release notes for GitHub (can hand-write, but `/g-release` surfaces a template)
    - Output: tag sha + release URL
11. Register both skills in `commands/g-forge.md` (now `commands/g-proof.md` per Wave 1)

## Premortem

| Risk | Scenario | Mitigation |
|------|----------|-----------|
| **Rename complexity** | Git history rewrites break tools / users have local clones with old remote | GitHub auto-redirects; old clone URLs still work (remote remains reachable); no force-push needed. Field installs just run `/g-update` to resync. |
| **Marketing mismatch** | README promises "G-Proof standard" but the product isn't fully strict | Be precise: the gate **enforces** review, period. Discipline is in the structure, not the model. No overselling. |
| **Version jump culture shock** | Users see v2.x → v3.0.0 and expect major features | Changelog + announcement make clear: v3.0.0 is a rebranding (semver major), same feature set, same enforcement model. |
| **GIF/diagram missing** | Gate demo can't be captured or rendered | Fallback: detailed text sequence (commit attempt → blocked → review → approved → commit succeeds) with ASCII art; less visceral but functional. |
| **Skills integration gap** | `/g-changelog` + `/g-release` poorly integrated into workflow | Anchor them to real usage: the only time you run `/g-release` is at milestone close, when approval is already there; no orphaned skills. |
| **Downstream breakage** | Field installs run `/g-update` but plugin registry doesn't sync correctly | Test `/g-update` on this machine against the renamed plugin; confirm it pulls the right version. |

## Rollback Plan

Rename is essentially one-way (GitHub manages it), but if needed before push:
1. Rename repo back to `hllrm/g-forge`
2. Revert plugin.json + marketplace.json name/version
3. Revert all `g-proof` → `g-forge` in CLAUDE.md + skills
4. Discard README and return to current version
5. No git history touched; cleanup is swift

Once pushed to `main` + released: rollback requires a v2.3.0 re-release (old name, old version), which we won't do — announce the new name and move forward.
